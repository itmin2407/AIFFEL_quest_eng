# Chapter 13 체크포인트

## 1. ML 테스트가 "다크 소울"이라 불리는 이유 (전통 SW와 비교 3가지)

### 1) 데이터 의존성
- 전통 SW: 입력이 제한적이고, 명세대로만 들어온다고 가정하고 테스트를 짠다.
- ML: 입력 분포가 넓고, 실제 프로덕션 데이터 분포를 완전히 아는 것이 불가능해서, “테스트 케이스를 모두 나열”하는 방식이 성립하지 않는다.

### 2) 비결정성·확률적 동작
- 전통 SW: 같은 입력 → 같은 출력(결정적), pass/fail 기준이 명확하다.
- ML: 같은 데이터라도 seed, 학습 상태, 샘플링 전략에 따라 조금씩 다른 결과가 나와 “정답 하나”를 기준으로 테스트하기 어렵다.

### 3) 품질 기준의 모호함
- 전통 SW: 사양 위반인지 아닌지 명확하게 판단 가능하다.
- ML: metric(accuracy, F1 등)이 연속값이고, “얼마 이상이면 충분한가?”, “특정 세부 케이스는 얼마나 중요하게 볼 것인가?” 같은 의사결정이 필요해서, 테스트 설계가 곧 제품·비즈니스 의사결정과 얽힌다.

---

## 2. 테스트 수트를 "분류기"로 봤을 때 오탐(False Alarm)의 위험

테스트 수트를 “이 변경(PR)이 안전한가?”를 예측하는 분류기라고 보면:

- 오탐(False Alarm) = 실제로는 문제 없는 코드인데 테스트가 실패했다고 판단하는 경우이다.

### 위험 사례 예시
- 예: 간단한 문구 수정, 로그 레벨 변경 등 위험이 매우 낮은 PR도, flaky test 때문에 자꾸 실패하면, 개발자들이 “어차피 깨지는 테스트”라며 테스트를 무시하거나 local에서만 돌리고 병합하는 문화가 생긴다.
- 결과적으로 진짜 버그를 잡는 신호도 무시되기 시작해, 배포 안정성이 급격히 떨어지는 “소리 큰 테스트의 소년이 양치기 소년이 되는” 상황이 된다.

## 3. 암기 테스트(overfit_batches=1) 실패 시 의심할 5가지 원인과 디버깅

암기 테스트: 아주 작은 배치(예: 1~16개)를 넣고, 해당 배치에 대해 loss가 0 가까이 떨어질 때까지 학습되는지 확인하는 테스트.

### 1) 모델 용량 부족 / 잘못된 아키텍처
- 원인: 레이어 수·hidden dim이 지나치게 작거나, 출력 레이어 설정이 label 차원과 안 맞음.
- 디버깅: batch size=1, learning rate 크게, 학습 반복 수 충분히 준 뒤, gradient norm·loss curve를 로그로 찍어 급격히 줄어드는지 확인. 모델 구조를 단순한 MLP/linear로 바꿔서라도 암기가 되는지 먼저 본다.

### 2) 손실 함수 / 레이블 매핑 오류
- 원인: CrossEntropy인데 label이 one-hot, label 인덱스와 vocab/클래스 매핑이 어긋남.
- 디버깅: 한 샘플의 logits와 label을 직접 프린트해서 손실이 0이 되려면 어떤 값이 나와야 하는지 수동 계산해본다. label dtype, shape, 값 범위(int vs float)도 확인한다.

### 3) 옵티마이저/학습 루프 버그
- 원인: `optimizer.step()`을 안 호출, grad를 잘못 초기화, 모델을 `eval()`로 두고 있음, 학습 루프에서 `loss.backward()` 누락 등.
- 디버깅: 동일 샘플로 여러 step을 돌려 파라미터가 실제로 바뀌는지 확인하고, grad norm을 log로 남긴다. `training_step` 안의 손실/역전파 순서를 최소 repro 코드로 축소해서 점검한다.

### 4) 정규화/규제 과도 (dropout, weight decay, gradient clipping 등)
- 원인: dropout이 0.5 이상, weight decay가 너무 커서 파라미터가 수렴하지 못함.
- 디버깅: 암기 테스트에서는 `dropout=0`, `weight_decay=0`, `clip_grad=None` 등으로 꺼보고 다시 돌려본다.

### 5) 데이터 전처리/augmentation 문제
- 원인: 입력과 label을 맞지 않게 셔플, augment가 매 step마다 label과 불일치, normalization에서 NaN/Inf 발생.
- 디버깅: `overfit_batches=1`일 때는 augment를 끄고(raw input 고정), dataloader에서 실제로 모델로 들어가는 tensor와 label을 직접 로깅해 눈으로 확인한다.

## 4. 데이터 기댓값 테스트에서 "느슨한 경계" 원칙

- 데이터 기댓값 테스트는 “컬럼 값의 분포/범위/결측 비율이 예상 범위 안에 있는가?”를 보는 테스트이다.
- 현실의 데이터는 항상 노이즈와 계절성, 신규 캠페인 등 다양한 이유로 조금씩 변하기 때문에, **경계를 너무 타이트하게 잡으면, 정상적인 변화까지 전부 실패로 간주**하게 된다.

### 문제점
- 자주 깨지는 테스트는 곧 무시된다(오탐 문제).
- 새로운 비즈니스 이벤트(예: 신규 국가 확장, 새로운 카테고리 런칭)를 전부 “에러”라고 보고하여, 팀이 빠르게 실험하기 어렵다.

### 결론
- 예: 평균 클릭률 0.1±0.02 → 대신 0.05~0.2 범위처럼 여유 있게 잡고, 장기적으로 baseline을 업데이트하는 식으로 관리하는 것이 좋다.

## 5. 행동 테스트: 불변성 vs 방향성 (예시)

텍스트 분류(욕설 탐지) 프로젝트를 예로 들면:

### 1) 불변성 테스트 (invariance test)
- 정의: 특정 변환을 해도 출력이 바뀌지 않아야 하는 테스트.
- 예시: 문장에 이모지나 공백을 추가해도 욕설 여부가 바뀌지 않아야 한다.
  - 입력 A: “너 진짜 이상하다”
  - 입력 B: “너 진짜 이상하다 😂”
  - 기대: `model(A) == model(B)`.

### 2) 방향성 테스트 (directional test)
- 정의: 특정 변환을 하면 출력이 “예상된 방향으로” 변해야 하는 테스트.
- 예시: 문장에 강한 욕설 표현을 추가하면 toxicity score가 올라가야 한다.
  - 입력 A: “너 오늘은 별로야”
  - 입력 B: “너 오늘 진짜 X같이 별로야”
  - 기대: `toxicity(B) > toxicity(A)`.

# Chapter 14 체크포인트

## 1. 왜 ML에서는 "Fast"가 "Right"보다 먼저인가?

- 전통 SW에서는 correctness를 먼저 확보하고, 그 다음 성능을 튜닝하는 패턴이 일반적이다.
- ML에서는 “Right(고성능, 정교한 모델·튜닝)”을 위해선 많은 실험을 해야 하고, 그 실험 속도를 결정하는 것이 infra/코드의 효율(=Fast)이기 때문에, **Fast하지 않으면 아예 Right에 도달할 기회가 줄어든다**.
- 즉, 실험 회전율(experiment velocity)이 곧 최종 품질을 결정하므로, Fast가 Right의 선결 조건이 된다.

## 2. OOM 오류 4단계 전략 & Adam 메모리 사용 이유

### OOM(Out Of Memory) 해결 4단계

1. 가장 쉬운 것부터 줄이기  
   - batch size 감소, sequence length/이미지 해상도 축소, `num_workers` 조절 등.

2. 모델/옵티마 단순화  
   - 작은 모델로 교체, layer 수/hidden dim 줄이기, Adam → SGD 등 메모리 적은 옵티마 사용.

3. 정교한 기법 적용  
   - Gradient checkpointing, mixed precision(fp16/bf16), offloading(optimizer states를 CPU/디스크로).

4. 하드웨어 스케일 업/아웃  
   - 더 큰 GPU, multi-GPU(DataParallel, DDP), pipeline/텐서 병렬 등 인프라 확장.

### Adam이 파라미터의 3~4배 메모리를 쓰는 이유

- 각 파라미터 w에 대해 1차 모멘트 m(평균 gradient), 2차 모멘트 v(gradient 제곱의 이동 평균)를 따로 저장한다.
- 따라서 weight(1배) + m(1배) + v(1배) (+ FP16에서 FP32 master weight까지 고려하면 3~4배 수준) 메모리가 필요하다.

## 3. 프로파일링 결과에 따른 최적화 우선순위

### 1) `get_train_batch`가 전체 시간의 60%

- 병목: 데이터 로딩/전처리/전송.

**우선 시도:**
- `DataLoader` `num_workers`/tensor pinning/배치 생성 구조 개선.
- I/O bound면: 데이터 캐시, 파일 포맷 변경(예: CSV → Parquet/LMDB), 사전 전처리.
- augmentation을 GPU로 옮기거나, 간소화/미리 생성.

### 2) `training_step`이 90%

- 병목: 모델 forward/backward.

**우선 시도:**
- mixed precision 사용, 작은 모델/efficient layer(Depthwise conv, FlashAttention 등)로 변경.
- unnecessary logging 제거, gradient checkpointing, 프로파일로 가장 느린 layer를 찾아 교체.

---

## 4. 스케일링 법칙: 작은 실험 → 직선 외삽 → 자원 추산

1. 작은 모델·작은 데이터로 여러 점(예: 파라미터 수 vs loss)을 찍어 본다.
2. 로그 스케일에서 loss가 parameter/data 크기에 대해 거의 직선처럼 감소하는 구간을 찾는다.
3. 이 직선을 바깥으로 연장(extrapolation)해서, “목표 성능에 도달하려면 대략 파라미터/데이터/연산량이 어느 정도 필요하겠다”를 거칠게 계산한다.

### 비즈니스 가치

- 무작정 거대 모델을 만들지 않고, **미리 비용(학습 시간/클러스터 크기/클라우드 비용)을 예측**할 수 있어, “이 프로젝트를 할 만한 투자 대비 효과가 있는가?”를 사전에 판단할 수 있다.
- 또한, 스케일링해도 성능 향상이 크지 않을 영역을 미리 알고, 그 돈을 다른 곳(데이터 품질 개선, UX)으로 돌릴 수 있다.

---

## 5. 성능 저하 진단표(오버피팅/언더피팅/분포 변화)

### 진단표 개념

- 오버피팅: train↑, val↓  
  - 처방: regularization 강화, 모델 축소, 데이터 증강, early stopping 등.

- 언더피팅: train/val 둘 다 낮음  
  - 처방: 모델 용량 증가, 학습 더 오래, feature 엔지니어링/프롬프트 개선.

- 분포 변화(데이터 쉬프트): 과거 val은 좋았는데 최근 prod 성능이 떨어짐  
  - 처방: 최근 데이터로 재학습, 도메인 적응, RAG 지식 업데이트.

### 예시: 로그 기반 이상 탐지 프로젝트

- train/val은 잘 맞았지만, 새 버전 서비스에서 로그 포맷·트래픽 패턴이 바뀌면서 알람 precision이 급락했다면 → 분포 변화 케이스.
- 적절한 처방: 최신 로그로 재학습, 새로운 필드 반영, 드리프트 모니터링 추가.

# Chapter 15 체크포인트

## 1. LLM 시스템을 Accuracy만으로 평가할 수 없는 이유 & 추가 평가 차원 5개

### 이유

- LLM은 출력 공간이 매우 크고, 하나의 “정답 문자열”만 있는 문제가 많지 않다.
- 또한, 안전성·정책 준수·UX 측면까지 포함해야 실제 서비스 품질을 평가할 수 있다.

### 추가 평가 차원 예시 5개

1. **정합성/사실성(Factuality)** – 지식 베이스와 일치하는지, 헛소리가 얼마나 적은지.
2. **안전성/유해성(Safety/Toxicity)** – 혐오·폭력·개인정보 노출 여부.
3. **지시 준수도(Instruction following)** – 프롬프트의 요구사항(형식, 언어, 스타일)을 얼마나 잘 지키는지.
4. **일관된 포맷/구조(Format adherence)** – JSON 스키마, 마크다운 규칙 등 계약된 형태를 얼마나 잘 따르는지.
5. **사용자 경험(UX) 지표** – 응답 시간, 길이 적절성, 사용자 만족도/재사용률 등.

---

## 2. LLM-as-a-Judge: 장점·한계(편향)·완화 방법

### 장점

- 사람 대신 대량 평가를 자동화할 수 있어, 모델/프롬프트/파이프라인 실험을 빠르게 비교 가능.
- BLEU·ROUGE 같은 단순 자동 메트릭보다 의미·발화 품질을 잘 반영할 수 있다.

### 한계(편향)

- 특정 스타일/표현을 선호하는 편향이 있어서, 다양한 정답 중 “LLM이 좋아하는 답”에 높은 점수를 줄 수 있다.
- 평가 대상 모델과 같은 계열/패밀리 모델을 쓰면 자기 집단 편향(self-preference)이 생길 수 있다.

### 편향 완화 방법 제안

- 서로 다른 LLM(모델/벤더·버전)이 섞인 다수결 평가를 사용하고, 일부 샘플은 꼭 사람 평가와 교차 검증해 “LLM 심판의 calibration curve”를 유지한다.

## 3. 입력 드리프트 / 출력 드리프트 / 성능 드리프트

- 입력 드리프트: 모델에 들어오는 **입력 데이터 분포 자체가 시간에 따라 달라지는 것** (예: 사용자 쿼리 유형 변화).
- 출력 드리프트: 같은 유형의 입력에 대해, 모델이 내놓는 **출력의 분포/스타일이 변하는 것** (예: 응답 길이가 점점 길어짐).
- 성능 드리프트: 골든 세트/실제 KPI 기준으로 **정답률·전환율 같은 성능 지표가 감소하는 것**.

### 예시 분석

- 로그 기반 이상 탐지나 추천 시스템이라면, 서비스 개편·사용자 층 변화로 **입력 드리프트**가 가장 먼저 일어날 가능성이 크다.
- LLM 챗봇이라면, 프롬프트/모델 버전 변경, RAG 지식 업데이트 등으로 **출력 드리프트**도 자주 발생할 수 있다.

## 4. "모델은 시간이 지나면 썩는다" – 코로나 사례 & 필요한 시스템

### 의미

- 세상(데이터 분포)이 바뀌는데 모델 파라미터는 그대로라서, 시간이 지나면 점점 현실을 못 따라가고 성능이 떨어진다는 뜻이다.

### 코로나 사례

- 코로나 이전에 학습한 수요 예측·이상 탐지·추천 모델은 “대면 활동, 출근 패턴”을 기반으로 학습되었는데, 팬데믹 이후 재택·온라인 주문 비중이 급증하면서, 예전 패턴이 통하지 않게 되었다.
- 이때 모델을 그대로 쓰면, 재고/배송/트래픽 예측이 크게 빗나가 “썩은 모델”이 된다.

### 필요한 시스템

- 지속적 모니터링(성능/드리프트) → 재학습 파이프라인(MLOps) → 롤백/AB 테스트를 지원하는 **자동/반자동 모델 라이프사이클 관리 시스템**이 필요하다.
- 즉, “학습은 한 번, 운영은 영원히”가 아니라, 주기적/이벤트 기반 재학습과 배포를 자동화하는 인프라가 필요하다.
